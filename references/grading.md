# 告警分级标准（Grading）

> 统一 P0-P3 定义，跨客户语义一致。所有五模式（monitor / audit / traffic / ir / remote）共享本表。SKILL.md 输出契约中的 `severity` 字段必须从本表取值。

---

## 一、四级定义

### P0 —— 紧急（真实命中且影响重大）

- **定义**：已确认或高度疑似攻击成功，对核心业务 / 数据 / 监管面有直接影响。
- **响应 SLA**：**立即**（≤ 15 分钟内介入，不等待下一个值守窗口）
- **通报对象**：现场带队 → 客户安全负责人 → （视情况）监管单位
- **典型场景**（≥5）：
  1. webshell 落地确认（路径 + 内容样本）
  2. fastjson / log4j / shiro 等反序列化 RCE 触发成功（返回了执行结果）
  3. SSH / RDP / 数据库管理面在演习期间陌生 IP 成功登录
  4. C2 心跳建立（主机持续外联陌生 IP，含 beacon 间隔特征）
  5. 主机失陷信号（新增 SUID 程序 / 计划任务 / 隐藏进程）
  6. 横向移动确认（同账号在多台主机间短时间内登录）
- **必动作**：升 IR 模式、保留快照、通报、止血

### P1 —— 高危（疑似真实命中或高价值线索）

- **定义**：攻击行为已发生，影响待评估，或高价值线索需要在小时级深挖。
- **响应 SLA**：**< 1 小时**
- **通报对象**：现场带队
- **典型场景**（≥5）：
  1. sqlmap / xray / nuclei 等工具发起的探测且 4xx 比例 < 50%（疑似命中）
  2. 敏感路径访问（`.git/config` / `.env` / `actuator/env`）返回 200
  3. SSH 暴破后**单次成功登录**，无后续命令
  4. 内部用户在非工作时段从异常 IP 登录
  5. 主机新增 root 等价账户但无后续动作
  6. WAF 拦截了高危 payload 但日志显示后端仍收到部分请求
- **必动作**：转 audit 深挖，2 小时内闭环

### P2 —— 中危（值得跟进的扫描或异常）

- **定义**：可观察到的探测 / 异常行为，未确认成功命中，作为情报积累。
- **响应 SLA**：**< 4 小时**
- **通报对象**：值守内部，日报体现
- **典型场景**（≥5）：
  1. 单 IP 短时间内大量 4xx（疑似目录爆破）
  2. 工具 UA（sqlmap / fscan / dirsearch）但 100% 4xx 失败
  3. 单账号多次密码错误（< 暴破阈值）
  4. 异常时段定时任务变更（但未发现持久化）
  5. 非业务端口偶发探测
- **必动作**：加入观察名单，关联升级触发即升 P1

### P3 —— 低危（误报概率高 / 噪音）

- **定义**：明显误报、合法扫描、健康检查等噪音。
- **响应 SLA**：**< 24 小时**（实际多为批量归类不处理）
- **通报对象**：仅值守内部归档
- **典型场景**（≥5）：
  1. 内部漏扫工具白名单 IP（如 `192.168.99.0/24` 的扫描器段）
  2. 健康检查 `/healthz` / `/ping` 返回 200
  3. 监控探针的固定 UA 与固定路径
  4. 搜索引擎爬虫（Googlebot / Bingbot，已核身份）
  5. 合规扫描时间窗内的内部扫描
  6. 业务正常异常（如开发联调期的 500）
- **必动作**：批量归档，每周复查一次白名单

## 二、响应 SLA 速查

| 级别 | 介入时限 | 通报时限 | 闭环时限 | 通报粒度 |
|---|---|---|---|---|
| P0 | ≤ 15 min | ≤ 30 min | ≤ 4 h | 现场带队 + 客户安全负责人 + 监管（按规则） |
| P1 | ≤ 1 h | ≤ 2 h | ≤ 8 h | 现场带队 |
| P2 | ≤ 4 h | 日报 | ≤ 24 h | 值守内部 |
| P3 | ≤ 24 h | 日报归档 | 周报复查 | 值守内部 |

通报粒度按客户合同 / 监管要求覆盖；本表为最低标准。

## 三、升降级规则

### 自动升级触发条件

| 条件 | 升级动作 |
|---|---|
| 同 IP 在 5 分钟内命中 ≥ 3 类不同告警规则 | 当前级别 +1（最高升到 P0） |
| 同账号在 10 分钟内于 ≥ 3 台主机出现登录 / 操作 | +1，同时打 `lateral` 标签 |
| 同 UA 命中工具指纹 + 后续敏感路径返回 200 | 直接升 P1 |
| 暴破账号在 1 小时内首次成功登录 | 直接升 P0（疑似爆破成功） |
| 命中 P3 但 IP 在历史 IOC 库内 | 升 P2 复核 |

### 降级条件（必须保守）

| 条件 | 降级动作 |
|---|---|
| 确认源 IP 在内部漏扫白名单且时段在合规扫描窗口 | 降到 P3 |
| 工具 UA 但全部 4xx 且 24 小时无后续 | 降到 P3 |
| 误报已二次复核（≥ 2 名值守独立判定） | 按复核结论 |

**保守原则**：降级需 ≥ 2 名值守独立确认；升级单值守即可触发。

## 四、误报概率与加权公式

输出契约中的 `false_positive_prob` 字段（0.0-1.0）按如下公式计算后给出 final_severity 建议：

```
final_severity_score = base_severity_score
                     + corroboration_boost
                     - fp_penalty

# 映射回级别：
# score >= 9  → P0
# score >= 7  → P1
# score >= 5  → P2
# score <  5  → P3
```

### 各项含义

| 项 | 含义 | 取值范围 | 示例 |
|---|---|---|---|
| `base_severity_score` | 命中规则的固有严重度 | 0-10 | webshell 落地命中 = 9；sqlmap UA = 4 |
| `corroboration_boost` | 关联证据加分 | 0-4 | 同 IP 5 分钟内 3 类告警 = +2；同账号横向 = +3 |
| `fp_penalty` | 误报扣分 | 0-5 | 源 IP 在白名单 = +5 减分；UA 是已知健康检查 = +4 |
| `final_severity` | 综合后建议级别 | P0-P3 | 按映射区间换算 |

**实施约定**：
- 公式仅用于"建议"，不替代人工研判；最终级别仍由值守标注。
- `false_positive_prob` 单独保存原始概率（建议从规则元数据 + 历史 FP/TP 比统计），不与上面 score 强耦合。
- 当 `fp_penalty >= 4` 时建议直接置为 P3，无论 base 多高（极强误报信号优先）。

## 五、跨客户一致性

不同甲方对"P0"的内部叫法可能不同（紧急 / 严重 / Critical / Sev1 等）。**Skill 内部统一使用 P0-P3**；输出报告时根据客户偏好做一次映射，映射表在客户接入时确认。

```yaml
# 映射示例（按客户保存到 customer_profile.yaml）
customer: <customer>
severity_map:
  P0: 紧急 / Sev1
  P1: 严重 / Sev2
  P2: 中等 / Sev3
  P3: 低 / Sev4
```

映射只影响最终面向客户的渲染，Skill 内部状态机、IOC、关联规则一律走 P0-P3。
